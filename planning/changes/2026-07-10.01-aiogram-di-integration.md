# Design: modern-di integration for aiogram

**Status:** design, pending maintainer approval
**Target:** new repo `modern-python/modern-di-aiogram`, package `modern_di_aiogram`

## Summary

Ship the official [modern-di](https://modern-di.modern-python.org) integration
for [aiogram](https://docs.aiogram.dev) **3.x**, following the core repo's
[Writing an integration](../../../modern-di/docs/integrations/writing-integrations.md)
contract along the **middleware + `@inject` decorator** path (aiogram has no
`Depends`-style seam — handlers receive values from a middleware `data` dict —
so this is the same family as `modern-di-aiohttp`/`-starlette`/`-typer`, not the
native-DI path taskiq used). One module (`modern_di_aiogram/main.py`) exposes
`setup_di`, `fetch_di_container`, `FromDI`, `inject`, and two connection
providers. An update-level outer middleware opens one `Scope.REQUEST` child
container per update; `@inject` (or `auto_inject=True`) resolves `FromDI` params
from it.

## Motivation

aiogram is the dominant async Telegram-bot framework and the second "Next up"
roadmap integration after taskiq — a new entrypoint class (bots) for the
"one wiring, every entrypoint" goal. Dishka has an aiogram integration; modern-di
has none.

## Verified facts (aiogram 3.29.1 + modern-di 2.27, spiked end-to-end)

Every non-obvious decision below was proven against installed aiogram, not
inferred:

- **Update-level `data` propagates to handlers.** Data set in
  `dp.update.outer_middleware` reaches whatever handler ultimately fires, so a
  single update-level middleware suffices — no per-observer registration.
- **`Update.event` gives the concrete event** (Message / CallbackQuery / …) and
  `Update.event_type` names it, *at update-level middleware time*. So the
  middleware can seed context with both `{Update: event, TelegramObject: event.event}`
  and expose the concrete event without registering per observer.
- **`@inject` must NOT use `functools.wraps`.** aiogram computes a handler's
  params via `inspect.unwrap(callback)` (`dispatcher/event/handler.py`), which
  follows `__wrapped__` back to the original function and **defeats a set
  `__signature__`**. The wrapper must instead copy `__name__/__qualname__/
  __doc__/__module__` by hand and set `__signature__` (Dishka does the same) —
  never `functools.wraps`. With that, aiogram reads the rewritten signature:
  DI params stripped, a keyword-only container param added, so aiogram passes
  the child container by name and never tries to fill the DI params itself.
- **`auto_inject` wraps handlers at startup.** A `dp.startup` callback walks
  `router.chain_tail` observers (skipping the `update` observer) and replaces
  each un-injected handler's `callback`/`params` with the injected version —
  the performant path (Dishka's `inject_router`), not a per-call middleware.
- **Lifecycle** rides `dp.startup` (→ `container.open`) and `dp.shutdown`
  (→ `container.close_async`); both are `EventObserver`s with `.register(...)`.
- **`Dispatcher.feed_update(bot, update)`** drives an update in-process (a fake
  `Bot(token="123456:AAA-…")` needs no network unless a handler calls it) — the
  test harness, analogous to taskiq's `InMemoryBroker`.

## Design

### Connection providers (kind → scope)

```python
aiogram_update_provider = providers.ContextProvider(aiogram.types.Update, scope=Scope.REQUEST)
aiogram_event_provider = providers.ContextProvider(aiogram.types.TelegramObject, scope=Scope.REQUEST)
_CONNECTION_PROVIDERS = (aiogram_update_provider, aiogram_event_provider)
```

`Update` and `TelegramObject` are distinct type keys (no `DuplicateProviderTypeError`);
the middleware seeds `{Update: update, TelegramObject: update.event}`.

### Container handoff (two named keys)

- `_ROOT_CONTAINER_KEY` — the root container, stored in the dispatcher's
  workflow-data (`dispatcher[_ROOT_CONTAINER_KEY] = container`), read by
  `fetch_di_container`.
- `_CHILD_CONTAINER_KEY` — the per-update child, stashed by the middleware in
  the per-update `data` dict; it is also the **keyword-only param name** the
  `@inject` wrapper adds, so aiogram fills it from `data` by name.

Keeping these on separate keys avoids the root (workflow-data) and child
(per-update data) colliding.

### `setup_di(dispatcher, container, *, auto_inject=False) -> Container`

```python
def setup_di(dispatcher, container, *, auto_inject=False):
    dispatcher[_ROOT_CONTAINER_KEY] = container
    container.add_providers(*_CONNECTION_PROVIDERS)
    dispatcher.startup.register(container.open)          # reopen-safe restart
    dispatcher.shutdown.register(container.close_async)
    dispatcher.update.outer_middleware(_DiMiddleware(container))
    if auto_inject:
        dispatcher.startup.register(functools.partial(_inject_router, dispatcher))
    return container
```

### Per-update child (`_DiMiddleware`)

```python
class _DiMiddleware(aiogram.BaseMiddleware):
    def __init__(self, container): self.container = container
    async def __call__(self, handler, event, data):
        child = self.container.build_child_container(
            scope=Scope.REQUEST, context={Update: event, TelegramObject: event.event})
        data[_CHILD_CONTAINER_KEY] = child
        try:
            return await handler(event, data)
        finally:
            await child.close_async()
```

`event` is the `Update` at update-level; `event.event` is the concrete event.
The child is closed in `finally` — including on the handler error path.

### `FromDI` marker + `inject` decorator (signature rewrite)

```python
@dataclasses.dataclass(slots=True, frozen=True)
class _FromDI(typing.Generic[T_co]):
    dependency: providers.AbstractProvider[T_co] | type[T_co]

def FromDI(dependency, /):  # noqa: N802 — stands in for a type
    return typing.cast(T, _FromDI(dependency))

def inject(func):
    di_params = _parse_inject_params(func)          # name -> _FromDI
    if not di_params:                                # nothing to inject
        func.__modern_di_injected__ = True
        return func
    sig = inspect.signature(func)
    visible = [p for n, p in sig.parameters.items() if n not in di_params]
    if _CHILD_CONTAINER_KEY not in sig.parameters:
        visible.append(inspect.Parameter(_CHILD_CONTAINER_KEY,
            kind=inspect.Parameter.KEYWORD_ONLY, annotation=Container))
    async def wrapper(*args, **kwargs):
        container = kwargs.pop(_CHILD_CONTAINER_KEY)
        resolved = {n: container.resolve_dependency(m.dependency) for n, m in di_params.items()}
        return await func(*args, **kwargs, **resolved)
    wrapper.__name__ = func.__name__; wrapper.__qualname__ = func.__qualname__
    wrapper.__doc__ = func.__doc__; wrapper.__module__ = func.__module__
    wrapper.__signature__ = sig.replace(parameters=visible)   # NOT functools.wraps
    wrapper.__modern_di_injected__ = True
    return wrapper
```

`resolve_dependency` dispatches provider-vs-type (inherits overrides, caching,
suggestions). The parse/rewrite helpers mirror `modern-di-aiohttp`.

### `auto_inject` (startup handler wrapping)

```python
def _inject_router(router):
    for sub in router.chain_tail:
        for observer in sub.observers.values():
            if observer.event_name == "update":
                continue
            for handler in observer.handlers:
                if not getattr(handler.callback, "__modern_di_injected__", False):
                    wrapped = aiogram.dispatcher.event.handler.HandlerObject(
                        callback=inject(handler.callback), filters=handler.filters, flags=handler.flags)
                    handler.callback = wrapped.callback
                    handler.params = wrapped.params
```

### Public API (`__all__`)

`setup_di`, `fetch_di_container`, `FromDI`, `inject`, `aiogram_update_provider`,
`aiogram_event_provider`.

## Non-goals

- **aiogram 2.x** — a different API; 3.x only.
- **aiogram-dialog** — a separate integration (its own roadmap item).
- **Async resolution** — resolution stays sync; only the container builder and
  finalizers are async.
- **Per-message nested scopes / per-observer registration** — the update is the
  unit of work; one REQUEST child per update.
- Any runtime dependency beyond `aiogram` and `modern-di`; a separate docs site.

## Testing

`Dispatcher.feed_update` with a fake `Bot`, 100%-coverage gate (`just test-ci`),
mirroring the sibling suites:

- `dependencies.py` — a `Group` with app- and request-scoped `Factory`
  providers plus one reading the event (`event: TelegramObject | None = None`,
  optional-with-default so construction-time `validate=True` passes while the
  real event still injects), proving context injection.
- `test_inject.py` — a handler with `@inject` resolving app-scoped, request-scoped,
  and event-derived values; feed an update, assert the handler saw them.
- `test_auto_inject.py` — the same via `setup_di(..., auto_inject=True)` with no
  per-handler decorator.
- `test_lifespan.py` — `dp.emit_startup()` opens the root, `dp.emit_shutdown()`
  closes it, a second cycle re-opens without `ContainerClosedError`;
  `fetch_di_container` returns the same instance.
- Error-path teardown — a handler that raises; assert the per-update child was
  still closed (a REQUEST provider finalizer ran).

## Risk

- **R1 — aiogram 3.x API drift.** `Update.event`, `router.chain_tail`,
  `HandlerObject`, and the `inspect.unwrap` behaviour are relied on. *Mitigation:*
  pin `aiogram>=3.0,<4` (ruled by maintainer). The implementation step installs
  the `3.0.0` floor and runs the suite to confirm those APIs exist at the low
  bound; if one landed later in 3.x, raise the floor to the lowest green version
  and report it. The `feed_update` integration test is the ongoing regression gate.
- **R2 — the `functools.wraps` trap.** aiogram's `inspect.unwrap` silently
  defeats `__signature__` if `__wrapped__` is set; the decorator would then leak
  DI params to aiogram (a "missing argument"/`KeyError`). Locked in above and
  guarded by the resolution tests, which fail loudly if it regresses.
- **R3 — `auto_inject` wraps at startup.** Handlers registered *after*
  `dp.emit_startup()` are not wrapped. Documented (register handlers before
  startup — the normal aiogram pattern); Dishka has the same constraint.
